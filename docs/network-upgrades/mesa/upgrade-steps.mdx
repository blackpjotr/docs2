# Mesa Upgrade Steps

:::warning Network Hard Fork — 2026-04-27 14:00 UTC

The Mesa preflight network will hard-fork on **2026-04-27 at 14:00 UTC**. A stop-slot release, **`4.0.0-preflight1-aaaa`**, is now available and is **mandatory** for every node operator. Nodes that are still on the pre-fork build (`4.0.0-preflight1-b649c79`) will fall off the network at the stop-transaction slot.

Upgrade to `4.0.0-preflight1-aaaa` as soon as possible. Once upgraded, the same binary will carry your node across the fork automatically.

:::

## Release summary

- **Pre-fork build (historical):** `4.0.0-preflight1-b649c79` — the original Mesa preflight release. Keeps running the current chain up to the stop slot, then stops producing and accepting blocks.
- **Stop-slot / hard-fork build (install this one):** `4.0.0-preflight1-aaaa` — contains the stop-slot configuration and the automode hard-fork logic. Runs the current chain before the fork and the new chain after it, with no binary swap required.

## Hard fork timeline

Unlike Berkeley, the Mesa upgrade uses **automode hard-forks**: a single binary (`4.0.0-preflight1-aaaa`) halts the old chain at the stop slot, performs the state transition automatically, and starts producing the new chain at the post-fork genesis timestamp.

| Milestone | Slot | Timestamp (UTC) | What happens |
|-----------|------|-----------------|--------------|
| Preflight genesis (historical) | 0 | 2025-11-27 13:00:00 | Original Mesa preflight chain start. |
| **Stop transaction slot** | 144 800 | **2026-04-27 10:00:00** | Block producers stop accepting new user transactions. |
| **Stop network slot** | 144 880 | **2026-04-27 12:00:00** | Network stops producing blocks. Stabilisation window ends. |
| **Auto hard fork trigger** (+80 slots) | 144 960 | **2026-04-27 14:00:00** | Hard fork executes automatically. |
| **Post hard fork genesis** | 0 (new era) | **2026-04-27 14:00:00** | New Mesa chain begins producing blocks. |

The 80-slot window between the stop network slot and the auto hard-fork trigger (~2 hours) is the stabilisation / bootstrap period — nodes can stay running, no operator action is required.

## What node operators must do

### Before 2026-04-27 10:00:00 UTC (stop transaction slot)

Every node operator must have upgraded to `4.0.0-preflight1-aaaa` before the stop transaction slot. See [Preflight Network](./preflight-network) for install instructions; simply use the new version tag when pulling Docker images or installing the Debian packages.

Quick reference:

- **Docker image tags:** `gcr.io/o1labs-192920/mina-daemon:4.0.0-preflight1-aaaa-bookworm-mesa` (and the matching `mina-archive` / `mina-rosetta` images).
- **Debian packages:** `mina-mesa=4.0.0-preflight1-aaaa`, `mina-archive-mesa=4.0.0-preflight1-aaaa`, `mina-rosetta-mesa=4.0.0-preflight1-aaaa` from the `preflight` channel of `unstable.apt.packages.minaprotocol.com`.

#### Block producers and SNARK workers

1. Pull / install `4.0.0-preflight1-aaaa` on every daemon and SNARK worker.
2. Keep the existing libp2p keypair and configuration — no new keys are required.
3. Restart nodes and confirm they report the new version via `mina client status`.
4. Keep block producers running through the stop transaction slot so the network maintains density during state finalisation.

#### Archive node and Rosetta operators

1. Install `mina-archive-mesa=4.0.0-preflight1-aaaa` (or pull the matching Docker image).
2. No schema changes ship with this release. If you have not yet run the Mesa archive upgrade script, follow [Archive Upgrade](./archive-upgrade) first — the script is idempotent and safe to run ahead of the fork.
3. Keep the archive running through the stabilisation window so the final pre-fork blocks land in Postgres.
4. After the fork, the same archive process will continue ingesting post-fork blocks — no restart required.

#### Exchanges and integrators

1. Upgrade all daemon and archive / Rosetta nodes to `4.0.0-preflight1-aaaa`.
2. Halt MINA deposits and withdrawals between the stop transaction slot (10:00 UTC) and the post-fork genesis (14:00 UTC). Transactions submitted in that window will not be part of the upgraded chain.
3. Re-enable user flows once your nodes have synced to the post-fork chain.

### Between 10:00 and 14:00 UTC on 2026-04-27 (state finalisation + fork)

- Leave nodes running; no manual action.
- Block production pauses at the stop network slot (12:00 UTC).
- The auto hard-fork fires at 14:00 UTC and the new chain starts immediately.

### After 2026-04-27 14:00:00 UTC (post-fork)

- Confirm your node is on the post-fork chain: `mina client status` should show the new genesis timestamp `2026-04-27T14:00:00Z`.
- Resume regular operations (transactions, staking, archive queries, Rosetta traffic).
- Report any issues on [Discord](https://discord.gg/minaprotocol) or the [Mina GitHub issue tracker](https://github.com/MinaProtocol/mina/issues).

## Rollback

If the fork needs to be aborted, the team will communicate on Discord and GitHub. Keep the pre-fork (`4.0.0-preflight1-b649c79`) Docker image / Debian package cached in case a rollback is announced, but do not downgrade proactively.
